§02 / Entry Points

Client Surfaces — Where the Prompt Is Born

Copilot is not a separate application; it is embedded into existing Microsoft 365 surfaces and inherits each surface's security posture — device compliance, app protection, DLP, conditional access. The surface determines authentication mechanics and local data handling, but every surface converges on the same in-boundary pipeline.1

W

Word, Excel, PowerPoint, OneNote

Desktop (Win/Mac) and web. Active document is the implicit grounding context — sent inline, not duplicated to OneDrive.

TLS 1.3 PRT-bound token
O

Outlook

Summarize / Draft surfaces. Mailbox content fetched via Graph at prompt-time under user identity.

TLS 1.3 E3 CA + MFA
T

Microsoft Teams

Pinned BizChat app + meeting recap. Transcripts and chat fetched from Graph; CA evaluated on signal.

TLS 1.3 E5 Endpoint DLP
C

m365.cloud · BizChat (web)

Standalone full-screen Copilot Chat. Uploaded files persist to OneDrive's "Microsoft Copilot Chat Files".2

TLS 1.3 in-tenant only
M

M365 Mobile App (iOS / Android)

App-protected container. Intune MAM prevents data leaving corporate scope — no device-FS persistence when MAM enforced.

TLS 1.3 Intune compliance
P

Plugins / Custom agent host

Per-plugin OAuth 2.0 consent. Highest-risk surface — admin App Governance review essential.

E3 weak gov E5 App-Gov full
✓ The convergence principle
Every surface — desktop, web, mobile, plugin host — converges on the same orchestrator → Graph → AOAI pipeline once inside the boundary. The strong implication for security teams: you do not need a "Copilot security strategy." You need to harden Microsoft 365 itself, and Copilot inherits that posture by construction.
§03 / End-to-End

The 22-Hop Data Flow — Prompt to Response

Microsoft's official architecture documents collapse the data flow into 4–5 narrative steps.1, 4 In practice, a single prompt traverses 20+ discrete control points. Below is the expanded walkthrough — every hop annotated with its enforced control and E3-vs-E5 strength.

1

User keystroke in surface

User types prompt in Word's Copilot pane, Outlook's "Summarize", BizChat, Teams, or mobile. Active document content captured by host app as implicit grounding context.

Control: App Protection Policy (MAM) E3 weak E5 + Endpoint DLP
2

Conditional Access pre-flight evaluation

Entra ID evaluates user, device compliance, location, sign-in risk, application context. Microsoft publishes an explicit "Protect AI with Conditional Access" template for Copilot.1

Control: Entra Conditional Access + MFA E3 strong E5 + risk-based (Entra P2)
3

Token issuance & PRT binding

Entra issues access token bound to device's Primary Refresh Token where supported. MFA enforced if not already satisfied. Token includes user + tenant + scope claims.

Control: OAuth 2.0 + OpenID Connect, MFA Zero Trust E3 strong
4

Encrypted egress over Microsoft Global Network

Prompt + context payload leaves device over TLS 1.2/1.3, enters closest of 150+ MGN peering points, routes to tenant's Copilot regional endpoint.5

Control: TLS, certificate pinning, MGN routing TLS 1.3 E3 strong
5

Service boundary entry

Request crosses into Microsoft 365 service boundary — tenant-scoped trust zone subject to M365's contractual security commitments.5 Copilot is a "core online service" under Microsoft Product Terms.

Control: M365 service boundary, EDP contract E3 strong
6

Orchestrator receives the prompt

Copilot Orchestrator — explicitly described by Microsoft as "not an AI but a coordinator" — receives input and begins Retrieval-Augmented Generation.5 Plans which knowledge sources, plugins, agents, and skills to invoke.

Control: System metaprompt, RAI safety filters E3 strong (built-in)
7

Prompt classification & safety filter (input side)

Azure AI Content Safety + Prompt Shields scan prompt for prompt injection, jailbreak attempts, harmful content, and policy violations before any data fetch begins.

Control: Azure AI Content Safety, Prompt Shields E3 strong
8

Conversation context & memory recall

Orchestrator retrieves prior turns from user's Copilot chat history (per-user, tenant-scoped). User can review or delete; chats subject to Purview retention.1

Control: Purview retention, eDiscovery E3 search only E5 search + purge
9

Grounding query → Microsoft Graph

Orchestrator performs hybrid keyword + vector retrieval against Graph index. Critical: query executes in user identity context — Graph enforces ACLs as gatekeeper.5, 6

Control: Identity-based ACL enforcement Zero Trust E3 strong
10

Semantic Index lookup

Per-user vectorized semantic representations of tenant content broaden search beyond exact keyword to conceptual meaning.6 Vectors honor user-level permissions; data stays in tenant; not used to train foundation LLMs.

Control: Per-user index, tenant residency E3 strong
11

Sensitivity label & encryption check

For every candidate item, Purview sensitivity labels and Azure RMS encryption are evaluated. To use a labeled+encrypted item, user must have EXTRACT + VIEW usage rights.7

Control: Purview Sensitivity Labels + RMS E3 manual labels E5 auto-label + container
12

SharePoint / OneDrive oversharing controls

Restricted SharePoint Search (RSS), Restricted Content Discoverability (RCD), Restricted Access Control (RAC), and DAG reports limit which sites surface in Copilot, without changing user permissions.7, 8

Control: SharePoint Advanced Management (SAM) E3 bundled w/ Copilot E5 strong
13

External data — connectors & agents

Synced connectors read from Graph index (ACL-enforced); federated connectors (preview, MCP-based) fetch in real time; agents may call external APIs over OAuth.9, 10

Control: Per-connector ACL, app-governance, OAuth scope review E3 limited gov E5 full App-Gov
14

Web grounding (optional, admin-toggled)

If web grounding enabled, orchestrator extracts keywords and queries Bing Index via Bing API.5 Bing Index does not learn from this interaction. Admins can disable per tenant or per Copilot.

Control: Web access policy toggle, Bing API isolation E3 strong
15

Grounding payload assembly

Orchestrator assembles the grounded prompt: original user prompt + retrieved Graph snippets + connector results + web results + system metaprompt + RAI guardrails.

Control: Metaprompt injection, RAI grounding rules E3 strong
16

Inference call → Azure OpenAI (M365-dedicated)

Grounded prompt sent to Azure OpenAI Service instance exclusively reserved for Microsoft 365 and inside the M365 service boundary.5 Microsoft does not train foundation LLMs on customer data; AOAI does not retain customer data.

Control: Dedicated AOAI, no-train, EU Data Boundary E3 contractual EDP
17

LLM generation

Foundation model (default GPT-4 family; Anthropic Claude opt-in for some Cowork & Studio agents under same EDP commitments) generates response from grounded prompt.

Control: Inference within service boundary E3 strong
18

Output safety filter (response side)

Generated response screened by Azure AI Content Safety for harmful content, hallucinated PII, prompt-injection-induced exfiltration, grounding consistency.

Control: Azure AI Content Safety output filter E3 strong
19

Sensitivity-label inheritance

If new content was generated from labeled sources, highest-priority sensitivity label inherited automatically (where supported), and protection settings are enforced even when file is later moved outside the tenant.7

Control: Purview label inheritance E3 manual E5 automatic
20

Audit + DSPM for AI logging

Prompt, response, referenced content, metadata written to unified audit log and surfaced in Purview DSPM for AI dashboards. E5 unlocks full prompt+response visibility; E3 sees activity only.8, 11

Control: Purview Audit + DSPM for AI E3 activity only E5 full content view
21

DLP & Insider-Risk evaluation

Outbound response checked against DLP policies. On E5, Adaptive Protection from Insider Risk Management can dynamically tighten DLP based on risk score. Communication Compliance can flag risky Copilot interactions.8

Control: Purview DLP + Adaptive Protection E3 SP/Exch/OD only E5 +Teams +Endpts +Adaptive
22

Encrypted return + render

Response travels back over TLS 1.2/1.3 to originating surface and renders inline. Copilot Pages content stored in user-owned SharePoint Embedded containers; uploaded files persist in OneDrive's "Microsoft Copilot Chat Files".7

Control: TLS, Preservation Hold, retention TLS 1.3 E3 strong
§04 / Layer 2

Identity, Auth & Conditional Access

Copilot inherits Microsoft 365 identity. There is no Copilot-specific authentication. The same Entra ID account, MFA configuration, and Conditional Access policies that gate access to Outlook or SharePoint also gate Copilot.1

Auth Stack

OAuth 2.0 + OpenID Connect

  • Tokens issued by Entra ID, bound to device PRT where supported.
  • Token includes user object ID, tenant ID, scopes (Files.Read.All, Mail.Read, etc.) — Graph evaluates these per call.
  • Service-to-service hops (orchestrator → Graph → AOAI) use signed managed-identity tokens, not the user token directly.

E3 strongE5 strong

Conditional Access

Policy-driven gating

  • Standard CA policies (user, device, location, app) work on Copilot identically.
  • Microsoft Learn ships an explicit "Protect AI with Conditional Access" template.
  • E5 unlocks risk-based CA via Entra ID Protection P2 (sign-in risk, user risk).

E3 CAE5 + risk

Multi-Factor Auth

Phishing-resistant by preference

  • Copilot uses tenant's existing MFA — security defaults, per-user MFA, or CA-driven.
  • Microsoft recommends MFA enabled. With security defaults, MFA on by default.
  • FIDO2, Windows Hello, Authenticator app, certificate-based auth all supported.

E3 strongE5 strong

Device & App Posture

Intune compliance + MAM

  • Device compliance feeds CA — non-compliant device blocks Copilot.
  • App protection (MAM) wraps Copilot in mobile to prevent data leaving corporate container.
  • Endpoint DLP (E5) blocks copy/paste/save of Copilot output containing labeled data.

E3 compliance onlyE5 + Endpoint DLP

§05 / Layer 3

Orchestrator, Grounding & RAG

The Copilot Orchestrator is the brain of the system — but as Microsoft is careful to point out, it is not itself an AI; it is a deterministic coordinator that plans, dispatches, and composes calls across LLMs, the Graph, plugins, and agents.5 It runs the Retrieval-Augmented Generation (RAG) pipeline.

What it does

Per-prompt orchestrator workflow

  1. Receives prompt + active-document context from the surface.
  2. Runs system metaprompt + RAI guardrails through Azure AI Content Safety (input filter).
  3. Decides which knowledge sources to query (Graph, Semantic Index, web, plugins, agents).
  4. Issues parallel grounding queries — Graph search executes as the user, so ACLs are enforced at the index level.
  5. Assembles the grounded prompt: user prompt + retrieved snippets + plugin outputs + system metaprompt.
  6. Dispatches to the LLM (default: GPT-4 family in M365-dedicated AOAI; Anthropic Claude opt-in for some agents).
  7. Runs output through Content Safety again, applies sensitivity-label inheritance, returns to surface.
  8. Writes prompt + response + reference list to unified audit log and DSPM for AI.
▎ Hybrid retrieval
The orchestrator's Graph query is hybrid — combining traditional keyword/lexical search with vector (semantic) search.6 This is plain RAG, not fine-tuning. Your tenant data is the ingredients, never the recipe.
§06 / Layer 4

Microsoft Graph & Semantic Index

The Graph is the gatekeeper. Microsoft's exact framing: "The large language model can't decide to access content the user doesn't have access to, as it's constrained by the same mechanism that constrains the user in other scenarios outside of Copilot."5

Microsoft Graph

Identity-bound search index

  • Indexes SharePoint, OneDrive, Exchange, Teams, Loop, Planner, Yammer/Engage, plus external connector content.
  • Honors per-user ACLs at the index level — Copilot's query result is pre-filtered to what the user can see.
  • Backs Microsoft Search, profile cards, and Copilot uniformly.
Semantic Index for Copilot

Per-user vector embeddings

  • Vectorizes tenant content for conceptual similarity rather than keyword match.
  • Built per-user; respects existing permissions; does not affect storage quotas.
  • Vectors stay in tenant; not used to train any foundation LLM.6
⚠ The oversharing problem
Because the Graph faithfully enforces ACLs, Copilot will surface everything the user is technically permitted to see — including content they were never expected to discover (legacy "Everyone except External" sites, mis-shared HR drives, abandoned Teams). SharePoint Advanced Management's RSS / RAC / RCD policies and Purview sensitivity labels are how you fix this without revoking permissions. SAM E3 bundled labels E3 manual labels E5 auto

§08 / Layer 5

Azure OpenAI & the LLM Boundary

The LLM is hosted in an Azure OpenAI Service instance exclusively reserved for Microsoft 365 and inside the Microsoft 365 service boundary.5 This is the single most misunderstood part of the architecture.

Where the model runs

Inside the M365 service boundary

It is not OpenAI's infrastructure. It is not the public Azure OpenAI tenant your dev team uses. It is a dedicated, contractually segregated AOAI instance under Microsoft Product Terms.

Microsoft's contractual commitments: no training of foundation LLMs on customer data; AOAI does not retain customer data; EU Data Boundary applies.

Which models

GPT-4 family + Anthropic Claude (opt-in)

Default models for most Copilot features are GPT-4 / GPT-4o variants. Anthropic Claude available as a subprocessor under same Microsoft Enterprise data protections — Microsoft has formally onboarded Anthropic as a subprocessor for Microsoft Online Services.3

If your tenant uses Cowork or custom Studio agents that select Claude, the grounded prompt is sent to Anthropic's infrastructure under those EDP commitments.

▎ What never reaches the LLM
The raw Graph index does not. The semantic vectors do not. The user's full mailbox does not. Only the grounded prompt — selected snippets retrieved by the orchestrator under user-context ACLs — reaches the model. This is the architectural reason the boundary holds: even a fully-jailbroken LLM cannot exfiltrate what the orchestrator never gave it.

§10 / Cryptography

Encryption, Protocols & the EU Data Boundary

In transit

TLS 1.2 / 1.3 everywhere

  • Client → MGN: TLS 1.2/1.3, Perfect Forward Secrecy.
  • Service-to-service inside boundary: mTLS + managed-identity tokens.
  • Bing API for web grounding: TLS 1.3.
  • Plugin/agent calls: per-plugin TLS, OAuth 2.0 bearer.
At rest

Layered envelope encryption

  • Default: Microsoft service-managed keys, AES-256, BitLocker volume encryption.
  • Customer Key (CMK) — bring-your-own-key option for SharePoint, Exchange, OneDrive.
  • Double Key Encryption (DKE) — for the most sensitive content; Copilot will respect DKE-encrypted documents (cannot read them without user keys).
Data residency

EU Data Boundary & tenant region

  • Customer + pseudonymous personal data stays within EU Data Boundary for EU tenants.
  • Copilot interaction data follows tenant's primary region.
  • Multi-Geo capable for E3 + E5 customers using Multi-Geo SKU.
Compliance attestations

Inherited from M365 core

  • ISO 27001 / 27017 / 27018 / 27701
  • SOC 1, SOC 2 Type II, SOC 3
  • HIPAA, FedRAMP High (GCC, GCC-High, DoD)
  • GDPR, EU Data Boundary, C5, IRAP (ASD Blueprint alignment)

§12 / Threats

Threats, Mitigations & Hardening Priorities

The honest threat model. None of these are theoretical — Microsoft itself documents each in its deployment blueprints and the Anthropic-as-subprocessor disclosure.

Threat 01

Oversharing-driven exposure

Copilot honors permissions you set incorrectly. The most common Copilot incident is not a model failure — it's a 2018 SharePoint site shared with "Everyone except external users" suddenly being summarized in a Copilot answer to a junior employee.

Mitigation: SAM (RSS, RAC, RCD), DAG reports, sensitivity-label auto-application.

E3 partialE5 full

Threat 02

Prompt injection via untrusted content

Attacker emails the user a document with hidden instructions. User asks Copilot to summarize their inbox. Copilot reads the doc, executes the injected instructions, exfiltrates via plugin call.

Mitigation: Prompt Shields, App Governance over plugins, restrict agents to vetted connectors, treat email/ticket data as untrusted in custom actions.12

Threat 03

Plugin / agent data exfiltration

A user-consented plugin pipes prompt fragments to a third-party SaaS outside the boundary. Once consent is granted, every subsequent Copilot interaction that triggers the plugin leaks.

Mitigation: Disable user consent for non-verified plugins, App Governance (E5), audit OAuth grants quarterly.

Threat 04

Sensitive data in prompts

User pastes a confidential M&A draft into BizChat to "make it sound better." The draft is now in the user's chat history and OneDrive Copilot Chat folder.

Mitigation: DLP policies on prompts (E5 endpoint DLP), Adaptive Protection, user training, Comm Compliance flagging.

Threat 05

Compromised identity

Adversary phishes a privileged user. Once they have the session, Copilot will dutifully retrieve everything that user can see — at machine speed.

Mitigation: Phishing-resistant MFA (FIDO2), Entra ID Protection P2 risk-based CA, Privileged Identity Management for admin scopes.

Threat 06

Sensitivity-label inheritance gaps

Copilot generates new content from labeled sources, but the destination format doesn't support inheritance, or user manually downgrades the label.

Mitigation: Auto-labeling at rest (E5), restrict label downgrade with justification, monitor label-change events in audit.

✓ The 80/20 hardening priority
For any organization deploying Copilot at scale: (1) fix oversharing with SAM + sensitivity labels, (2) enforce phishing-resistant MFA + risk-based CA, (3) lock plugin/agent governance via App Governance, (4) turn on DSPM for AI and review weekly, (5) write retention policies that explicitly cover Copilot interactions. Everything else is decoration on top of these five.
